Introduction

Voice and style form the personality of your documentation. While content accuracy ensures correctness, voice determines whether readers can understand, trust, and act on that information.

This article explores:

  • Active vs. passive voice - When to use each and why Microsoft, Google, and Wikipedia differ
  • Readability formulas - Understanding Flesch-Kincaid scores and this repository‚Äôs 50-70 target
  • Sentence structure - Why 15-25 words per sentence improves comprehension

  • Practical examples - Before/after transformations showing principles in action

Prerequisites: Understanding of foundational documentation principles is helpful but not required.

Active vs. Passive Voice

Voice refers to the relationship between the subject and verb in a sentence. This seemingly simple grammatical choice profoundly affects clarity, tone, and reader engagement.

Definitions and Examples

Active voice: Subject performs the action

  • Structure: [Subject] [Verb] [Object]
  • Example: ‚ÄúThe developer writes the code.‚Äù
  • Emphasis: Who/what is doing something

Passive voice: Subject receives the action

  • Structure: [Object] [is/was] [Verb-ed] [by Subject]
  • Example: ‚ÄúThe code is written by the developer.‚Äù
  • Emphasis: What is being done

When to Use Active Voice

Clarity and directness - Active voice makes actors explicit

‚ùå Passive (unclear): > ‚ÄúThe API should be initialized before any requests are made.‚Äù

‚úÖ Active (clear): > ‚ÄúInitialize the API before you make any requests.‚Äù

Responsibility and agency - Active voice assigns accountability

‚ùå Passive (vague): > ‚ÄúAn error was encountered during deployment.‚Äù

‚úÖ Active (specific): > ‚ÄúThe deployment script encountered an error.‚Äù > Or better: ‚ÄúYou encountered an error during deployment.‚Äù

Brevity - Active constructions are typically shorter

‚ùå Passive (wordy - 11 words): > ‚ÄúThe configuration file must be updated by the administrator.‚Äù

‚úÖ Active (concise - 7 words): > ‚ÄúThe administrator must update the configuration file.‚Äù > Or: ‚ÄúUpdate the configuration file.‚Äù

When Passive Voice Is Acceptable

Actor is unknown or irrelevant

‚úÖ Appropriate passive: > ‚ÄúThe server was compromised at 3:00 AM.‚Äù > (Who compromised it is unknown)

Emphasis on the object, not the actor

‚úÖ Appropriate passive: > ‚ÄúPython was released in 1991.‚Äù > (Focus on Python‚Äôs history, not Guido van Rossum‚Äôs action)

Scientific or encyclopedic tone (Wikipedia preference)

‚úÖ Appropriate passive: > ‚ÄúThe data was collected over six months.‚Äù > (Scientific detachment, methodology focus)

Avoiding accusatory tone

‚úÖ Appropriate passive: > ‚ÄúThe file was deleted accidentally.‚Äù > (Less accusatory than ‚ÄúYou deleted the file accidentally‚Äù)

Before/After Examples

Example 1: Procedural instruction

‚ùå Passive and wordy: > ‚ÄúAfter the installation has been completed by the user, the configuration wizard will be launched automatically and the initial settings should be reviewed carefully before the application is started.‚Äù

‚úÖ Active and clear: > ‚ÄúAfter you install the software, the configuration wizard launches automatically. Review the initial settings before you start the application.‚Äù

Example 2: Error message

‚ùå Passive and vague: > ‚ÄúAn invalid parameter was provided.‚Äù

‚úÖ Active and specific: > ‚ÄúYou provided an invalid parameter. Check the API documentation for valid values.‚Äù

Example 3: Technical reference (passive acceptable)

‚úÖ Passive (appropriate for reference material): > ‚ÄúThe function is called when the event fires. Three arguments are passed: event type, timestamp, and payload.‚Äù

Example 4: Tutorial (active preferred)

‚úÖ Active (appropriate for learning): > ‚ÄúCall the function when the event fires. Pass three arguments: event type, timestamp, and payload.‚Äù

Detecting Passive Voice

Linguistic test: Can you insert ‚Äúby zombies‚Äù after the verb?

  • ‚ÄúThe code was written [by zombies]‚Äù ‚Üí Passive
  • ‚ÄúThe developer writes the code [by zombies]‚Äù ‚Üí Doesn‚Äôt work, so active

Automated detection: This repository‚Äôs validation system flags excessive passive voice through grammar analysis.

Readability Formulas Explained

Readability formulas quantify text complexity, helping ensure documentation matches audience capabilities.

Flesch Reading Ease Score

Formula: 206.835 - (1.015 √ó ASL) - (84.6 √ó ASW)

  • ASL = Average Sentence Length (words per sentence)
  • ASW = Average Syllables per Word

Score interpretation:

Score Range Difficulty Typical Reader Example Context
90-100 Very Easy 11-year-old Children‚Äôs books
80-89 Easy 13-year-old Young adult fiction
70-79 Fairly Easy 15-year-old Magazines
60-69 Standard 17-18 year-old General documentation
50-59 Fairly Difficult College level Technical documentation
30-49 Difficult College graduate Academic papers
0-29 Very Difficult Professional Legal, scientific journals

This repository‚Äôs target: 50-70

  • Rationale: Balances technical precision with accessibility
  • Lower bound (50): Allows necessary technical complexity
  • Upper bound (70): Ensures general technical audience comprehension

Why we validate Flesch scores:

From validation-criteria.md:

readability:
  flesch_reading_ease:
    target_range: [50, 70]
    minimum_acceptable: 40
    rationale: "Technical content requires precision, but must remain accessible"

Flesch-Kincaid Grade Level

Formula: (0.39 √ó ASL) + (11.8 √ó ASW) - 15.59

Score interpretation:

  • Score = US grade level required to understand the text
  • Example: Score of 10 = 10th grade reading level

This repository‚Äôs target: 9-10

  • Equivalent to Flesch Reading Ease 50-70
  • Indicates high school to early college reading level

Gunning Fog Index

Formula: 0.4 √ó [(ASL) + 100 √ó (Complex Words / Total Words)]

  • Complex Words = words with 3+ syllables

Interpretation: Years of formal education required

Typical ranges for technical documentation:

  • 8-10: Accessible technical writing
  • 11-14: Standard technical documentation
  • 15+: Dense academic/specialist writing

Why Readability Matters: Cognitive Load

Cognitive load theory explains that working memory has limited capacity (typically 7¬±2 ‚Äúchunks‚Äù of information).

Factors increasing cognitive load:

  • Long sentences (>30 words)
  • Complex vocabulary (multi-syllable technical terms)
  • Dense paragraph structure
  • Passive voice constructions
  • Abstract concepts without examples

Strategies reducing cognitive load:

  • Shorter sentences (15-25 words)
  • Familiar words when possible
  • Transitional phrases between ideas
  • Active voice (fewer words, clearer actors)
  • Concrete examples illustrating abstractions

Readability Formula Limitations

Formulas cannot measure:

  • Technical accuracy
  • Logical coherence
  • Appropriate level of detail
  • Cultural context
  • Visual design impact

They are indicators, not mandates:

‚úÖ Good use of readability scores: > ‚ÄúThis section has a Flesch score of 35 (difficult). Can we simplify the language or add examples?‚Äù

‚ùå Bad use of readability scores: > ‚ÄúThis section must have a Flesch score of exactly 60. Replace all multi-syllable words.‚Äù

Repository principle: Readability scores inform validation, but technical accuracy always takes precedence. See validation workflow.

Sentence Structure and Length

Sentence length directly affects comprehension. Research shows optimal sentence length for technical documentation is 15-25 words.

Why 15-25 Words?

Psychological research: Reading comprehension drops significantly beyond 25 words per sentence

  • Reader must hold more information in working memory
  • Clause relationships become harder to track
  • Readers may need to re-read for understanding

Practical testing: Major tech companies‚Äô analysis shows 15-25 words optimizes:

  • Reading speed
  • Comprehension accuracy
  • User satisfaction with documentation

Sentence Type Distribution

Effective technical writing mixes sentence types:

Simple sentences (1 independent clause)

  • Best for critical information
  • Example: ‚ÄúSave your work before closing the application.‚Äù
  • Use: 40-50% of sentences

Compound sentences (2+ independent clauses, coordinating conjunction)

  • Connect related ideas of equal importance
  • Example: ‚ÄúThe function returns an object, and the object contains three properties.‚Äù
  • Use: 20-30% of sentences

Complex sentences (1 independent + 1+ dependent clause)

  • Show relationships between ideas
  • Example: ‚ÄúWhen the user clicks the button, the form validates all input fields.‚Äù
  • Use: 20-30% of sentences

Compound-complex sentences (2+ independent + 1+ dependent)

  • Use sparingly, risk comprehension
  • Example: ‚ÄúWhen you initialize the API, it loads the configuration, and then it establishes a connection.‚Äù
  • Use: 0-10% of sentences

Before/After: Sentence Length Optimization

Example 1: Breaking up a 45-word monster

‚ùå Too long (45 words, Flesch ~35): > ‚ÄúThe authentication system provides OAuth 2.0 support which allows users to authenticate using their existing social media accounts including Facebook, Google, and Twitter, and the implementation follows the authorization code flow pattern to ensure security.‚Äù

‚úÖ Optimized (3 sentences, 15-16 words each, Flesch ~60): > ‚ÄúThe authentication system supports OAuth 2.0. Users can authenticate using Facebook, Google, or Twitter accounts. The implementation follows the authorization code flow for security.‚Äù

Example 2: Combining choppy short sentences

‚ùå Too choppy (4-6 words each, Flesch ~80 but immature tone): > ‚ÄúOpen the file. Click Save. The file saves. A confirmation appears.‚Äù

‚úÖ Optimized (combined, 18 words total, Flesch ~65): > ‚ÄúOpen the file and click Save. The file saves, and a confirmation message appears.‚Äù

Example 3: Balancing technical complexity

‚ùå Dense single sentence (52 words, Flesch ~25): > ‚ÄúTo configure the load balancer to distribute incoming HTTP requests across multiple backend servers while maintaining session affinity through cookie-based tracking and implementing health checks that automatically remove unresponsive servers from the rotation, follow these steps.‚Äù

‚úÖ Optimized (3 sentences, ~17 words each, Flesch ~55): > ‚ÄúThis section explains how to configure the load balancer for HTTP requests. The configuration includes session affinity via cookies and automatic health checks. Health checks remove unresponsive servers from rotation.‚Äù

Voice Guidelines: Microsoft vs. Google vs. Wikipedia

Different organizations prioritize different aspects of voice. Understanding these priorities helps you make informed decisions.

Microsoft Writing Style Guide Approach

Core philosophy: ‚ÄúWrite like you speak‚Äù

Voice characteristics:

  • Person: Second person (you) throughout
  • Tone: Conversational but professional
  • Contractions: Acceptable (‚Äúdon‚Äôt‚Äù not ‚Äúdo not‚Äù)
  • Sentence length: ‚ÄúShort sentences are powerful‚Äù

Examples from Microsoft documentation:

‚úÖ Conversational Microsoft style: > ‚ÄúYou can use Azure Functions to run code without managing servers. When an event triggers your function, it runs in a fully managed environment.‚Äù

Microsoft‚Äôs rationale:

  • Users feel directly addressed
  • Reduces formality barrier
  • Increases engagement with content
  • Mirrors how developers actually speak

Google Developer Documentation Approach

Core philosophy: ‚ÄúWrite for a global audience‚Äù

Voice characteristics:

  • Person: Second person (you)
  • Tone: Friendly but precise
  • Contractions: Avoided for international clarity
  • Sentence length: Very short, scannable

Examples from Google documentation:

‚úÖ Clear Google style: > ‚ÄúUse Cloud Functions to deploy code without managing servers. Cloud Functions runs your code when an event triggers it.‚Äù

Google‚Äôs distinctive elements:

  • No contractions (international readers)
  • Ultra-short sentences (mobile-first)
  • Product names as subjects (not ‚Äúit‚Äù)
  • Explicit over implicit

Google vs. Microsoft comparison:

Aspect Microsoft Google
‚ÄúYou can‚Äôt configure‚Ä¶‚Äù ‚úÖ Acceptable ‚ùå Use ‚ÄúYou cannot‚Ä¶‚Äù
‚ÄúIt automatically scales‚Äù ‚úÖ Acceptable ‚ùå Use ‚Äú[Service name] automatically scales‚Äù

Wikipedia Manual of Style Approach

Core philosophy: ‚ÄúEncyclopedic neutrality‚Äù

Voice characteristics:

  • Person: Third person only
  • Tone: Neutral, academic
  • Contractions: Never
  • Sentence length: Varies, completeness valued over brevity

Wikipedia rules:

  • ‚ùå ‚ÄúYou should back up your data‚Äù
  • ‚úÖ ‚ÄúUsers should back up their data‚Äù or ‚ÄúData should be backed up‚Äù
  • ‚ùå ‚ÄúWe recommend‚Ä¶‚Äù
  • ‚úÖ ‚ÄúExperts recommend‚Ä¶‚Äù or ‚ÄúAccording to [source]‚Ä¶‚Äù

When Wikipedia‚Äôs approach is appropriate:

‚úÖ For encyclopedic content: > ‚ÄúThe Model-View-Controller (MVC) pattern separates concerns into three interconnected components. The model manages data, the view displays the interface, and the controller handles user input.‚Äù

‚úÖ For historical/technical background: > ‚ÄúREST was introduced by Roy Fielding in his 2000 doctoral dissertation. The architectural style emphasizes stateless communication and resource-based interactions.‚Äù

‚ùå Not appropriate for procedural content: > ‚ÄúThe user should open the terminal and type the command. The system will display output.‚Äù > (Awkward for instructions)

Person Usage (First, Second, Third)

Person choice fundamentally affects tone and reader relationship with content.

First Person (I, We, Us)

Appropriate contexts:

‚úÖ Tutorials (plural ‚Äúwe‚Äù as guide): > ‚ÄúIn this tutorial, we‚Äôll build a REST API from scratch. We‚Äôll start with basic routing, then add authentication.‚Äù

‚úÖ Author‚Äôs note in explanation: > ‚ÄúI recommend starting with the simpler approach before attempting the optimized version.‚Äù

‚úÖ Open-source project documentation (community ‚Äúwe‚Äù): > ‚ÄúWe welcome contributions from developers worldwide. We review pull requests within 48 hours.‚Äù

Inappropriate contexts:

‚ùå Reference documentation: > ‚ÄúWe provide three authentication methods‚Ä¶‚Äù (system provides, not ‚Äúwe‚Äù)

‚ùå How-to guides: > ‚ÄúWe need to configure the settings‚Ä¶‚Äù (reader configures, not collective ‚Äúwe‚Äù)

Second Person (You, Your)

Appropriate contexts:

‚úÖ Procedural instructions: > ‚ÄúYou must install Node.js before you run the application.‚Äù

‚úÖ Conditional guidance: > ‚ÄúIf you need real-time updates, use WebSockets instead of HTTP polling.‚Äù

‚úÖ Troubleshooting: > ‚ÄúIf you see error 404, check your URL configuration.‚Äù

Inappropriate contexts:

‚ùå Encyclopedic content: > ‚ÄúYou can find REST in many modern APIs‚Ä¶‚Äù (encyclopedias don‚Äôt address readers)

‚ùå Reference definitions: > ‚ÄúThe function accepts three parameters you pass‚Ä¶‚Äù (reference describes, doesn‚Äôt instruct)

Third Person (User, Developer, System)

Appropriate contexts:

‚úÖ API reference: > ‚ÄúThe function returns null when the user provides an invalid parameter.‚Äù

‚úÖ Conceptual explanation: > ‚ÄúDevelopers choose NoSQL databases when the data model requires flexibility.‚Äù

‚úÖ System behavior: > ‚ÄúThe application validates input before processing the request.‚Äù

Inappropriate contexts:

‚ùå Step-by-step instructions: > ‚ÄúThe user should click the button and enter their password.‚Äù (second person clearer)

Repository Guidelines

Based on documentation.instructions.md:

Content Type Preferred Person Rationale

Explanation Third person + educational first plural (we) Neutral analysis + shared exploration

Tone and Register

Tone conveys attitude; register indicates formality level. Both should match audience expectations and content purpose.

Tone Spectrum for Technical Documentation

Formal ‚Üê ‚Üí Conversational

Tone Example Appropriate Context
Formal ‚ÄúOne must ensure data persistence prior to system termination.‚Äù Academic papers, standards documents

Conclusion

Voice and style fundamentally shape documentation effectiveness. Key takeaways:

Active voice matters - Use it 75-85% of the time for clarity, but embrace passive voice when appropriate (scientific tone, unknown actor, emphasis on object)

Readability is measurable - Flesch scores 50-70 and sentence lengths 15-25 words optimize technical comprehension without oversimplification

Style guides differ intentionally - Microsoft favors conversational warmth, Google emphasizes global clarity, Wikipedia requires neutral encyclopedic tone - choose based on audience and content type

Person consistency matters - Second person (you) for instructions, third person for reference, first person plural (we) sparingly for tutorials

Tone should match purpose - Professional for how-to guides, austere for reference, warmer for tutorials, neutral for explanation

Common pitfalls are avoidable - Watch for person mixing, nominalizations, excessive hedging, anthropomorphizing, and buried verbs

This repository synthesizes principles - Validation system enforces voice standards while allowing appropriate flexibility
