This article synthesizes the latest GitHub Copilot, Visual Studio Code (VS Code) and Visual Studio documentation, along with community best practices, to provide clear guidance on naming and organizing Copilot-related files.

# üìÅ Use the `.github` folder for team-shared prompts and instructions

GitHub Copilot looks in specific subfolders of `.github/` to find <mark>prompt files, instructions and (in VS Code) custom agent definitions</mark>.

| Path                                | Purpose | 
|-------------------------------------|---------|
| <mark>**`.github/prompts/`**</mark>             | <mark>**Workspace prompt files.**</mark> Each `*.prompt.md` file here becomes a slash-command (`/promptName` in VS Code) or a hashtag command (`#promptName` or `#prompt:<prompt-name>` in Visual Studio) when using Copilot Chat. This folder is recognized by VS Code and **Visual Studio 2022 (version 17.10+)**. Older versions of Visual Studio (before 17.10) do not support prompt files. <mark>**Note that prompt files must use the `.prompt.md` extension**</mark>; <mark>**there is no official support for a plain `.prompt` extension**</mark>. The `.prompt` extension (without `.md`) is not recognized by Copilot in VS Code or Visual Studio. Each file can start with YAML front matter specifying metadata (`name`, `description`, `agent`, `model`, `tools`, etc.) followed by the Markdown content of the prompt. |
| <mark>**`.github/copilot-instructions.md`**</mark> | <mark>**Repository-level custom instructions.**</mark> This file provides guidance on how Copilot should behave for your project (coding standards, tool preferences, naming conventions, etc.). <mark>It's automatically applied to all chat requests when the "custom instructions" feature is enabled</mark>, and is supported by both VS Code and Visual Studio 17.10+. |
| <mark>**`.github/instructions/`**</mark>         | <mark>**Path- or language-specific instructions.**</mark> Store one or more `.instructions.md` files here. Each file uses YAML front matter fields like `applyTo` to specify globs (e.g., `**/*.cs`, `docs/**/*.md`) that determine when the instructions should be applied. Supported in both VS Code and Visual Studio 17.10+. |
| <mark>**`.github/agents/`**</mark>               | <mark>**Custom agents for VS Code / Copilot CLI**</mark>. In VS Code 1.106+ and Copilot CLI, `.agent.md` files here define specialized personas (e.g., "planning agent," "test specialist"). Each file contains YAML frontmatter to declare allowed `tools`, a `description`, optional `handoffs` for workflow transitions, `model` selection, and a Markdown body describing the agent's behavior. Legacy `.chatmode.md` files are still recognized but can be migrated using VS Code's Quick Fix action. <mark>**Note:** This is different from `AGENTS.md` files (see below)</mark>. <br><br>**v1.107+ Features:** <mark>**Claude Skills support**</mark> for extended agent capabilities. Agents can execute in three contexts: <mark>**local**</mark> (interactive), <mark>**background**</mark> (work tree isolated), or <mark>**cloud**</mark> (GitHub PRs). See [How GitHub Copilot Uses Markdown and Prompt Folders](./02-getting-started/01.00-how_github_copilot_uses_markdown_and_prompt_folders.md#-appendix-d-unified-agent-architecture-v1107) for details. |
| <mark>**`.github/skills/`**</mark>               | <mark>**Agent Skills for VS Code 1.107+ (Preview).**</mark> Skills are **portable, resource-rich workflows** that bundle instructions with templates, scripts, and examples. Each skill lives in its own subdirectory containing a `SKILL.md` file with YAML frontmatter (`name`, `description`) plus optional `templates/`, `examples/`, `scripts/`, and `checklists/` folders. Skills follow the [agentskills.io](https://agentskills.io/) open standard and work across VS Code Chat, Copilot CLI, and GitHub Coding Agent. Unlike prompts (on-demand tasks) or instructions (file-pattern rules), <mark>skills are discovered based on their `description` and can reference bundled resources</mark>. See [üéØ Agent Skills](#-agent-skills-vs-code-1107-preview) section for details. |
| <mark>**`.github/prompt-snippets/`**</mark>      | <mark>**Reusable context fragments.**</mark> Markdown files here are **not** indexed as slash commands. They serve as building blocks to include in prompts via `#file:.github/prompt-snippets/filename.md`. Use for repeated context blocks (architecture summaries, checklists, domain glossaries). See [üì¶ Supporting Assets](#-supporting-assets-snippets-and-templates) section. |
| <mark>**`.github/templates/`**</mark>            | <mark>**Starting point files for new resources.**</mark> Input templates (`.md` files) that you copy and customize when creating new prompts, agents, instructions, or skills. Examples: `prompt-template.md`, `agent-template.md`, `skill-template.md`. Templates include version numbers and changelogs for tracking updates. See [üì¶ Supporting Assets](#-supporting-assets-snippets-and-templates) section. |
| **`AGENTS.md`** (anywhere in repo)              | <mark>**Agent instructions for Copilot Coding Agent.**</mark> You can create one or more `AGENTS.md` files stored anywhere in the repository. The nearest file in the directory tree takes precedence. Alternatively, you can use `CLAUDE.md` or `GEMINI.md` in the repository root. These files provide instructions for AI agents working on your codebase, supported in VS Code, Visual Studio, and GitHub.com. <mark>**Note:** `AGENTS.md` provides instructions to agents, while `.agent.md` files define custom agent personas</mark>. |
| **`.github/copilot/`**               | **Community convention for shared context.** Some teams create a `.github/copilot/` folder to gather prompts and domain knowledge (e.g. `knowledge/`, `examples/`) in one place. <mark>Only a subfolder named `prompts/` under `.github` is recognized by default</mark>; other subfolders under `.github/copilot` serve as shared documentation and must be referenced manually in prompts or instructions. |

> **Note about `.prompt.yml` files**: GitHub also supports `.prompt.yml` or `.prompt.yaml` files for storing prompts intended for other GitHub AI tools (such as GitHub Models).  
These YAML-based prompt files are **not used by GitHub Copilot Chat** in VS Code or Visual Studio.  
For Copilot Chat, you must use the `.prompt.md` Markdown format.



### Naming Guidelines for Prompt Files

1. <mark>**Consistent naming:**</mark> Use lowercase with hyphens or dots (e.g., `create-react-form.prompt.md`) to reflect the task or role. <mark>The filename or the YAML `name` becomes the slash/hashtag command</mark>.
2. <mark>**Single purpose:**</mark> Each prompt file should perform a distinct, cohesive task (e.g., "generate unit tests" or "review API security"). Avoid mixing unrelated tasks in one prompt.
3. **Descriptive metadata:** Include a `description` in the YAML header. VS Code and Visual Studio display this text when listing the prompt in the Chat UI. You can also set fields like `agent` (ask, edit, agent mode), `model`, `tools`, or `argument-hint` to refine prompt execution.

# üíº Organizing Local Context in `.copilot` (Optional)

If you want Copilot to provide more project-aware suggestions, consider creating a <mark>`.copilot/`</mark> directory at your repository root. While this is **not an official requirement**, it's a community-recommended way to store detailed project documentation, domain concepts, architecture diagrams and other reference materials. <mark>Copilot's semantic search will index the contents of `.copilot/context/`</mark>, making it more likely to find relevant documents when answering questions about your codebase.

A typical `.copilot/context/` folder might include:

- `dataschemas/` ‚Äì JSON or database schema docs. 
- `apis/` ‚Äì internal and external API contracts. 
- `patterns/` ‚Äì coding conventions (e.g., error handling, logging, testing guidelines). 
- `workflows/` ‚Äì business processes and flows. 
- `guidelines/` ‚Äì coding standards, architecture principles, security patterns. 
- `images/` and `examples/` ‚Äì diagrams, charts and reference code.

<mark>These files do **not** create slash commands; instead, they enrich the context Copilot can draw upon</mark>.

# üë§ User Prompts in VS Code (Personal Prompt Library)

In addition to workspace prompts, <mark>**VS Code supports user-scope prompt files**</mark> that are available across all your projects. These personal prompts are stored in your VS Code profile directory:

- **Windows**: `%APPDATA%\Code\User\prompts\` (stable) or `%APPDATA%\Code - Insiders\User\prompts\` (Insiders)
- **macOS**: `~/Library/Application Support/Code/User/prompts/`
- **Linux**: `~/.config/Code/User/prompts/`

User prompt files:
- Appear in Copilot Chat as slash commands (e.g., `/my-personal-prompt`) across all workspaces
- Do **not sync to your team via Git** (they're stored locally in your profile)
- Can be shared across your own machines using <mark>**VS Code's Settings Sync**</mark> feature
- Follow the same `.prompt.md` format and YAML frontmatter as workspace prompts

<mark>**Visual Studio does not currently support user-scope prompt files**</mark>; it only recognizes workspace-level prompts in `.github/prompts/`.

# üí° Additional Tips

- <mark>**Centralize general rules**</mark> in `.github/copilot-instructions.md`, and use more granular `.instructions.md` files for narrower scopes (e.g. language-specific or directory-specific rules). Note that custom instructions must be enabled to be applied in Visual Studio or VS Code.
- **JetBrains IDE support:** Prompt files are now supported in JetBrains IDEs (IntelliJ, PyCharm, etc.). You can create prompt files via the settings page under Tools > GitHub Copilot > Customizations, or manually in `.github/prompts/`. Use `/promptName` in chat to invoke them.
- **Link to project docs** within your prompts or instructions using Markdown links, or use `#fetch` and `#file` references in VS Code to import remote or local content into Chat. (These tool references currently work in VS Code Chat; Visual Studio may not support every tool yet.)
- <mark>**Iterative improvement**</mark>: Test prompts and update them based on feedback. Copilot's responses are non-deterministic; small tweaks often change the quality of output. Encourage team members to contribute and refine prompts.

By following these naming and organization practices‚Äîand understanding where each file type is supported‚Äîyou can ensure that GitHub Copilot operates consistently across your team's tools and versions.

# üéØ Agent Skills (VS Code 1.107+ Preview)

<mark>**Agent Skills**</mark> are a **Preview feature** introduced in VS Code 1.107 that enable <mark>**portable, resource-rich workflows**</mark> bundling instructions with templates, scripts, and examples. Unlike prompts (on-demand slash commands) or instructions (file-pattern rules), skills are <mark>**discovered based on their description**</mark> and can reference bundled resources following the open [agentskills.io](https://agentskills.io/) standard.

## Skills vs. Other Resource Types

| Resource Type | Trigger | Scope | Bundled Resources |
|---------------|---------|-------|-------------------|
| **Prompts** (`.prompt.md`) | Explicit `/command` or `#command` | Single task | ‚ùå None |
| **Instructions** (`.instructions.md`) | File pattern match (`applyTo`) | Always-on rules | ‚ùå None |
| **Agents** (`.agent.md`) | Explicit `@agent` mention | Persona with tools | ‚ùå None |
| <mark>**Skills**</mark> (`SKILL.md`) | Description-based discovery | Complex workflow | ‚úÖ Templates, scripts, examples |

## Directory Structure

Each skill lives in its own subdirectory under `.github/skills/`:

```
.github/skills/
‚îú‚îÄ‚îÄ article-review/
‚îÇ   ‚îú‚îÄ‚îÄ SKILL.md                    # Required: Instructions + metadata
‚îÇ   ‚îú‚îÄ‚îÄ templates/                  # Optional: Reusable output templates
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ review-summary.md
‚îÇ   ‚îú‚îÄ‚îÄ checklists/                 # Optional: Structured checklists
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ validation-checklist.md
‚îÇ   ‚îî‚îÄ‚îÄ examples/                   # Optional: Real-world examples
‚îÇ       ‚îî‚îÄ‚îÄ sample-review.md
‚îú‚îÄ‚îÄ webapp-testing/
‚îÇ   ‚îú‚îÄ‚îÄ SKILL.md
‚îÇ   ‚îú‚îÄ‚îÄ templates/
‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ test-scaffold.ts
‚îÇ   ‚îî‚îÄ‚îÄ scripts/                    # Optional: Automation scripts
‚îÇ       ‚îî‚îÄ‚îÄ setup-test-env.sh
```

## SKILL.md Frontmatter

The `SKILL.md` file requires YAML frontmatter with two essential fields:

```yaml
---
name: article-review                  # Required: lowercase, hyphens, max 64 chars
description: >                        # Required: max 1024 chars
  Comprehensive article review workflow for technical documentation.
  Use when reviewing markdown articles, validating structure, 
  checking reference classifications, or preparing content for publication.
---
```

### Name Field Rules

| ‚úÖ Valid | ‚ùå Invalid | Reason |
|----------|-----------|--------|
| `article-review` | `Article Review` | No uppercase/spaces |
| `webapp-testing` | `test` | Too generic |
| `k8s-deploy` | `a-skill-name-that-exceeds-the-64-character-limit-check` | Too long |

### Description Field Formula

**Structure:** `[What it does] + [Technologies] + "Use when" + [Scenarios]`

```yaml
# ‚úÖ Good - specific with use cases
description: >
  Generate test suites for React components using Jest and RTL.
  Use when writing unit tests, debugging failures, or setting up test infrastructure.

# ‚ùå Bad - vague, no technologies, no scenarios
description: Helps with testing
```

## Progressive Loading System

Skills use a <mark>**three-level loading system**</mark> for efficient token usage:

| Level | What Loads | When | Token Impact |
|-------|------------|------|--------------|
| **1. Discovery** | `name` + `description` from YAML | Always indexed | ~50-100 tokens/skill |
| **2. Instructions** | SKILL.md body content | Prompt matches description | ~500-1500 tokens |
| **3. Resources** | Templates, examples, scripts | Copilot references them | Loaded on-demand |

This means your `description` field is critical‚Äîit determines whether the skill gets activated and its full instructions loaded.

## SKILL.md Body Content

After the YAML frontmatter, include structured instructions:

```markdown
---
name: article-review
description: >
  Comprehensive article review workflow for technical documentation.
  Use when reviewing markdown articles or preparing content for publication.
---

# Article Review Skill

## Purpose

Perform comprehensive quality reviews of technical documentation articles.

## When to Use

Activate this skill when:
- **Reviewing articles**: "Review this article for publication readiness"
- **Checking structure**: "Validate the structure of this markdown file"
- **Quality assessment**: "Assess the quality of this technical documentation"

Do NOT use this skill for:
- Grammar checking (use grammar-review prompt instead)
- Code review (use code-review skill instead)

## Workflow

### Step 1: Structure Validation

Check that the article includes all required sections:
- [ ] Title (H1 heading at the start)
- [ ] Table of Contents (for articles > 500 words)
- [ ] References section with classified sources

### Step 2: Content Quality Checks

Review for:
- [ ] Active voice usage
- [ ] Proper markdown formatting
- [ ] Code examples with language identifiers

## Templates

- [Review Summary Template](./templates/review-summary.md)
- [Validation Checklist](./checklists/validation-checklist.md)
```

## When to Use Skills vs. Prompts

**Use Skills when:**
- ‚úÖ Workflow requires **bundled resources** (templates, scripts, examples)
- ‚úÖ Task is **complex with multiple steps** and validation phases
- ‚úÖ Same workflow needs to work across **VS Code, CLI, and Coding Agent**
- ‚úÖ Team needs **portable, self-contained packages**

**Use Prompts when:**
- ‚úÖ Task is **simple and self-contained** (single action)
- ‚úÖ No additional resources needed beyond the prompt text
- ‚úÖ Explicit invocation via `/command` is preferred

# üì¶ Supporting Assets: Snippets and Templates

Beyond the core prompt, instruction, and agent files, GitHub Copilot ecosystems often include <mark>**prompt snippets and templates**</mark> to streamline development workflows. Understanding when to use each asset type‚Äîand where to store them‚Äîhelps teams build reusable, discoverable resources.

## Prompt Snippets

**<mark>Prompt snippets</mark>** are <mark>**reusable text fragments**</mark> designed to be inserted into chat conversations or combined with other prompts. Unlike full `.prompt.md` files, snippets are not standalone commands; they serve as building blocks for larger prompts or documentation.

### When to Use Prompt Snippets

| Use Case | Example |
|----------|---------|
| **<mark>Repeated context blocks</mark>** | Standard project architecture descriptions, technology stack summaries |
| **<mark>Common instructions</mark>** | Code style reminders, testing requirements, security checklist items |
| **<mark>Boilerplate questions</mark>** | "Explain the error handling approach", "Document the API contract" |
| **<mark>Domain vocabulary</mark>** | Business-specific terminology, acronym definitions, concept explanations |

### Where to Store Snippets

**<mark>Recommended location</mark>:** <mark>`.github/prompt-snippets/`</mark> or `.copilot/snippets/`

```
.github/
‚îú‚îÄ‚îÄ prompt-snippets/
‚îÇ   ‚îú‚îÄ‚îÄ architecture-context.md
‚îÇ   ‚îú‚îÄ‚îÄ security-checklist.md
‚îÇ   ‚îú‚îÄ‚îÄ testing-standards.md
‚îÇ   ‚îî‚îÄ‚îÄ domain-glossary.md
```

**Snippet file characteristics:**
- ‚ùå **No YAML frontmatter** - Snippets are raw content, not executable commands
- ‚ùå **Not indexed as slash commands** - Must be manually referenced
- ‚úÖ **Composable** - Multiple snippets can be combined in one conversation
- ‚úÖ **Lightweight** - Short, focused fragments (typically < 50 lines)

### How to Reference Snippet Files

GitHub Copilot **does not have automatic "include" syntax**. To use snippet files, reference them explicitly:

**Method 1: File Reference in Chat (VS Code)**
```markdown
Please review this code for security issues.

#file:.github/prompt-snippets/security-checklist.md
```
‚úÖ **Best for:** Interactive chat conversations, ad-hoc queries  
‚úÖ **Guarantees:** Content is definitely included in context

**Method 2: Markdown Link (Unreliable)**
```markdown
Please review this code following our [security checklist](.github/prompt-snippets/security-checklist.md).
```
‚ö†Ô∏è **Note:** Copilot may or may not automatically read linked files  
‚ùå **Not recommended** for critical content that must be included

**Method 3: Copy-Paste When Executing**
```markdown
Please review this code for security issues.

## Security Checklist
[Manually paste security-checklist.md content here when running prompt]
```
‚úÖ **Best for:** Ensuring snippet content is definitely included  
‚úÖ **Use when:** `#file` reference isn't working reliably

### Snippets vs. Embedded Content

**When to use snippet files** (reusable across prompts):
- ‚úÖ Content is used by **multiple prompts**
- ‚úÖ Shared **team knowledge** (architecture, standards, checklists)
- ‚úÖ Content needs **separate maintenance** from prompts
- ‚úÖ Changes should **propagate** to all prompts using it

**When to embed content directly** (single prompt use):
- ‚úÖ Content is **specific to one prompt** only
- ‚úÖ Content is **not reused** elsewhere
- ‚úÖ Prompt should be **self-contained**
- ‚úÖ No need for **separate file management**

**Example: Embedded content** (NOT using snippets):
````markdown
---
name: security-review-api-only
description: Security review focused specifically on REST API endpoints
---

Review this API code for security issues using this checklist:

## API-Specific Security Checklist
- [ ] Input validation on all parameters
- [ ] SQL injection prevention (parameterized queries)
- [ ] Authentication on protected routes
- [ ] Rate limiting configured
- [ ] CORS properly configured
[... content embedded directly, not shared with other prompts ...]
````

**Example: Using snippet file** (reusable):
```markdown
# In chat or prompt:
Review this code for security issues.

#file:.github/prompt-snippets/security-checklist.md
```

## Templates

**Templates** come in two flavors with different purposes:

### 1. Generated Templates (<mark>Output Templates</mark>)

<mark>**Definition:**</mark> Structures that prompts generate as output (e.g., code scaffolds, documentation skeletons, configuration files).

**Examples:**
- React component template (what a "generate component" prompt creates)
- API endpoint structure (what a "create REST endpoint" prompt produces)
- Test file layout (what a "scaffold tests" prompt outputs)

**Storage:** Reference these in prompt instructions, but don't store as separate files unless needed for consistency.

**Example in prompt file:**
````markdown
---
name: create-react-component
description: Generate a React component with TypeScript
---

Create a functional React component following this structure:

```tsx
import React from 'react';
import styles from './${componentName}.module.css';

interface ${componentName}Props {
  // Props here
}

export const ${componentName}: React.FC<${componentName}Props> = (props) => {
  return (
    <div className={styles.container}>
      {/* Component implementation */}
    </div>
  );
};
```

Include PropTypes and a basic test file.
````

### 2. Reusable Templates (<mark>Input Templates</mark>)

<mark>**Definition:**</mark> Starting points for creating new prompt files, agent files, or instruction files‚Äîtemplates you copy and fill in.

**Examples:**
- Blank `.prompt.md` template with standard YAML fields
- Agent definition template with common tools and handoffs
- Instruction file template with `applyTo` pattern examples

**Recommended location:** <mark>`.github/templates/`</mark> or `.copilot/templates/`

```
.github/
‚îú‚îÄ‚îÄ templates/
‚îÇ   ‚îú‚îÄ‚îÄ prompt-template.md
‚îÇ   ‚îú‚îÄ‚îÄ agent-template.md
‚îÇ   ‚îú‚îÄ‚îÄ instruction-template.md
‚îÇ   ‚îî‚îÄ‚îÄ README.md  # Explains when to use each template
```

**Example: `.github/templates/prompt-template.md`**
```yaml
---
name: [prompt-name]
description: [Brief description of what this prompt does]
agent: [ask|edit|agent]  # Optional: specify agent mode
model: [Model name]       # Optional: Claude Sonnet 4, GPT-4, etc.
tools: [tool1, tool2]     # Optional: limit available tools
argument-hint: [hint]     # Optional: help text for arguments
---

# [Prompt Title]

[Clear description of the task this prompt performs]

## Context

[Any background information Copilot needs]

## Instructions

1. [Step-by-step instructions]
2. [Be specific and actionable]

## Output Format

[Describe the expected output structure]

## Example

[Optional: provide an example of the desired result]
```

## Decision Framework: Snippet vs. Template vs. Full Prompt vs. Skill

| Asset Type | Purpose | Reusability | Discoverability | Typical Size |
|------------|---------|-------------|-----------------|--------------|
| **<mark>Prompt Snippet</mark>** | Context fragment for composition | High (included in multiple prompts) | Low (not indexed) | < 50 lines |
| **<mark>Generated Template</mark>** | Output structure definition | Medium (referenced in prompts) | Medium (via prompt docs) | Varies |
| **<mark>Input Template</mark>** | Starting point for new files | Low (copy-once) | Low (team documentation) | ~50 lines |
| **<mark>Full Prompt</mark>** | Standalone executable task | High (team-wide command) | High (slash/hashtag command) | 50-500 lines |
| **<mark>Agent Skill</mark>** | Complex workflow with resources | High (portable across platforms) | High (description-based discovery) | SKILL.md + bundled resources |

## Organization Best Practices

### 1. Clear Separation

Don't mix snippets, templates, skills, and prompts in the same folder:

‚îú‚îÄ‚îÄ SKILL.md               # Wrong location for skills!

‚úÖ GOOD: Clear folder structure
.github/
‚îú‚îÄ‚îÄ prompts/               # Executable prompt files

‚îú‚îÄ‚îÄ templates/             # Starting point files
‚îÇ   ‚îî‚îÄ‚îÄ prompt-template.md

    ‚îî‚îÄ‚îÄ article-review/

### 2. Document Usage

Include a `README.md` in each folder explaining:
- What files are stored there
- How to use them (copy, include, reference)
- When to create new files vs. using existing ones

### 3. Version Control for Templates

When you update an input template, <mark>**existing files created from that template are not automatically updated**</mark>. Consider:
- **Documenting template versions** in the template file itself
- **Creating migration guides** when templates change significantly
- **Using prompts to standardize** existing files when templates evolve

Example:
````markdown
---
# Template Version: 2.1
# Last Updated: 2025-12-25
# Changelog: Added 'tools' and 'model' fields, removed deprecated 'mode' field
---
````

### 4. Snippet Composition Patterns

**Pattern 1: Inline inclusion via `#file`** (VS Code only)
```
Please analyze this code. #file:.github/prompt-snippets/security-checklist.md
```

**Pattern 2: Manual copy-paste**
```markdown
Review the following for security issues:

[Content of security-checklist.md pasted here]
```

**Pattern 3: Reference in prompt YAML**
**Pattern 4: Markdown link** (may work, less reliable)
```markdown
Follow the guidelines in [security-checklist.md](.github/prompt-snippets/security-checklist.md).
```

## Common Patterns

### Snippets for Domain Context
```
.copilot/snippets/
‚îú‚îÄ‚îÄ business-rules.md      # Company-specific policies
‚îú‚îÄ‚îÄ architecture-overview.md  # System architecture summary
‚îî‚îÄ‚îÄ tech-stack.md          # Technology choices and versions

**Use case:** Include in multiple prompts requiring domain knowledge without duplicating content.

### Templates for Consistency

‚îú‚îÄ‚îÄ feature-prompt.md      # Template for feature implementation prompts
‚îú‚îÄ‚îÄ review-prompt.md       # Template for code review prompts
‚îî‚îÄ‚îÄ refactor-prompt.md     # Template for refactoring prompts
```

**Use case:** Ensure all team members create prompts with consistent structure.

### Generated Templates in Prompts

## Key Takeaways

- **Snippets** = Reusable context fragments included in other prompts
- **Generated Templates** = Structures that prompts output (code scaffolds, docs)
- **Input Templates** = Starting points for creating new prompt/agent/instruction/skill files
- **Full Prompts** = Standalone executable commands with YAML frontmatter
- **Agent Skills** = Portable workflows bundling SKILL.md with templates, scripts, and examples (VS Code 1.107+)

Store snippets and templates in dedicated folders, document their usage, and version them when they change. This organization ensures your team can easily find, reuse, and maintain supporting assets alongside core prompt files.

# ‚ö†Ô∏è Common Mistakes

Even with clear guidance on file organization, teams frequently make mistakes that reduce discoverability, create maintenance burdens, or cause confusion. Here are the most common organizational pitfalls and how to avoid them.

## Mistake 1: Mixing File Types in the Same Folder

**Problem**: Storing prompts, snippets, templates, and documentation in the same folder makes it unclear which files are executable commands versus reference materials.

**‚ùå Bad example:**
```
.github/prompts/
‚îú‚îÄ‚îÄ create-component.prompt.md     # Executable prompt
‚îú‚îÄ‚îÄ security-checklist.md          # Is this a prompt or a snippet?
‚îú‚îÄ‚îÄ template.md                    # What kind of template?
‚îú‚îÄ‚îÄ architecture-notes.md          # Documentation?
‚îî‚îÄ‚îÄ README.md
```

**‚úÖ Solution: Separate by type with clear folder structure:**
```
.github/
‚îú‚îÄ‚îÄ prompts/                       # Only executable .prompt.md files
‚îÇ   ‚îî‚îÄ‚îÄ create-component.prompt.md
‚îú‚îÄ‚îÄ prompt-snippets/               # Reusable fragments
‚îÇ   ‚îî‚îÄ‚îÄ security-checklist.md
‚îú‚îÄ‚îÄ templates/                     # Starting point files
‚îÇ   ‚îî‚îÄ‚îÄ prompt-template.md
‚îî‚îÄ‚îÄ docs/                          # Team documentation
    ‚îî‚îÄ‚îÄ architecture-notes.md
```

## Mistake 2: Inconsistent Naming Conventions

**Problem**: Team members use different naming styles, making prompts hard to discover and remember.

**‚ùå Bad examples:**
```
.github/prompts/
‚îú‚îÄ‚îÄ CreateReactComponent.prompt.md        # PascalCase
‚îú‚îÄ‚îÄ generate_unit_tests.prompt.md         # snake_case
‚îú‚îÄ‚îÄ ReviewAPI.prompt.md                   # Abbreviated, unclear
‚îî‚îÄ‚îÄ refactor-code-for-performance.prompt.md  # Too verbose
```

**‚úÖ Solution: Establish team convention (kebab-case recommended):**
```
.github/prompts/
‚îú‚îÄ‚îÄ create-react-component.prompt.md      # Clear, concise, kebab-case
‚îú‚îÄ‚îÄ generate-unit-tests.prompt.md         # Consistent pattern
‚îú‚îÄ‚îÄ review-api-security.prompt.md         # Full words, not abbreviated
‚îî‚îÄ‚îÄ refactor-performance.prompt.md        # Descriptive but concise
```

**Naming guidelines:**
- Use **kebab-case** (lowercase with hyphens)
- Start with **action verb** (`create`, `generate`, `review`, `refactor`)
- Include **domain/technology** when relevant (`react`, `api`, `database`)
- Keep **under 5 words** for command brevity

## Mistake 3: Wrong Extension or Location

**Problem**: Using incorrect file extensions or storing files in unrecognized locations prevents Copilot from discovering them.

**‚ùå Bad examples:**
```
.github/
‚îú‚îÄ‚îÄ copilot/prompts/                   # Wrong: Copilot doesn't scan here
‚îÇ   ‚îî‚îÄ‚îÄ my-prompt.prompt.md
‚îú‚îÄ‚îÄ prompts/
‚îÇ   ‚îú‚îÄ‚îÄ another-prompt.prompt          # Wrong: Missing .md extension
‚îÇ   ‚îî‚îÄ‚îÄ yet-another.md                 # Wrong: Missing .prompt part
‚îî‚îÄ‚îÄ my-instructions.md                 # Wrong: Should be copilot-instructions.md
```

**‚úÖ Solution: Use official paths and extensions:**
```
.github/
‚îú‚îÄ‚îÄ prompts/                           # Correct location
‚îÇ   ‚îî‚îÄ‚îÄ my-prompt.prompt.md            # Correct: .prompt.md extension
‚îú‚îÄ‚îÄ copilot-instructions.md            # Correct: exact filename
‚îî‚îÄ‚îÄ instructions/
    ‚îî‚îÄ‚îÄ python-rules.instructions.md   # Correct: .instructions.md extension
```

**Critical rules:**
- Prompt files: **`.github/prompts/*.prompt.md`** (not `.prompt` alone)
- Repository instructions: **`.github/copilot-instructions.md`** (exact name)
- Path-specific instructions: **`.github/instructions/*.instructions.md`**
- Agents (VS Code): **`.github/agents/*.agent.md`**

## Mistake 4: No README or Documentation

**Problem**: New team members don't know which prompts exist, what they do, or when to use them.

**‚ùå Bad example:**
```
.github/prompts/
‚îú‚îÄ‚îÄ p1.prompt.md
‚îú‚îÄ‚îÄ p2.prompt.md
‚îú‚îÄ‚îÄ p3.prompt.md
‚îî‚îÄ‚îÄ p4.prompt.md
```

No explanation of purpose, no usage examples, no maintenance notes.

**‚úÖ Solution: Add README.md in each folder:**
```markdown
<!-- .github/prompts/README.md -->
# Team Prompts

## Available Prompts

| Command | Purpose | When to Use |
|---------|---------|-------------|
| `/create-react-component` | Generate functional React component | Creating new UI components |
| `/generate-unit-tests` | Create Jest tests for selected code | After writing new functions |
| `/review-api-security` | Security audit for API endpoints | Before deploying API changes |

## Usage

1. Open Copilot Chat
2. Type `/` to see available prompts
3. Select prompt or type `/promptName`
4. Provide required context (selection, arguments)

## Adding New Prompts

See `.github/templates/prompt-template.md` for the standard structure.
```

## Mistake 5: Orphaned or Outdated Files

**Problem**: Old prompts remain in the repository long after they're obsolete, creating confusion and clutter.

**‚ùå Bad example:**
```
.github/prompts/
‚îú‚îÄ‚îÄ create-class-component.prompt.md       # Obsolete: Team uses hooks now
‚îú‚îÄ‚îÄ use-moment-js.prompt.md                # Obsolete: Migrated to date-fns
‚îú‚îÄ‚îÄ generate-flow-types.prompt.md          # Obsolete: Migrated to TypeScript
‚îî‚îÄ‚îÄ review-webpack-config.prompt.md        # Obsolete: Using Vite now
```

**‚úÖ Solution: Regular maintenance with deprecation markers:**
````markdown
---
# create-class-component.prompt.md
name: create-class-component
description: "‚ö†Ô∏è DEPRECATED: Use /create-react-component instead (functional components)"
---

**This prompt is deprecated.**

Our team migrated to functional components with hooks.
Use `/create-react-component` instead.

This file will be removed on 2025-03-01.
````

**Maintenance checklist:**
- [ ] Quarterly review of all prompt files
- [ ] Mark deprecated prompts with `‚ö†Ô∏è DEPRECATED` in description
- [ ] Set removal date in prompt body
- [ ] Announce deprecations in team channels
- [ ] Remove after grace period (1-2 months)

## Mistake 6: User Prompts for Team-Wide Tasks

**Problem**: Creating prompts in user-scope folder when they should be shared with the entire team.

**‚ùå Bad example:**
```
# Developer creates prompt in:
%APPDATA%\Code\User\prompts\review-api-security.prompt.md

# Problem: Only available on that developer's machine
# Team can't use it, can't improve it, can't maintain it
```

**‚úÖ Solution: Workspace prompts for team tasks, user prompts for personal workflows:**
```
# Team-shared prompts (version controlled):
.github/prompts/
‚îú‚îÄ‚îÄ review-api-security.prompt.md      # Everyone needs this
‚îú‚îÄ‚îÄ generate-unit-tests.prompt.md      # Team standard

# Personal prompts (developer's profile folder):
%APPDATA%\Code\User\prompts/
‚îú‚îÄ‚îÄ my-notes-format.prompt.md          # Personal note-taking style
‚îî‚îÄ‚îÄ my-commit-message.prompt.md        # Personal commit preferences
```

**Decision framework:**
- **Workspace prompts** (`.github/prompts/`) ‚Üí Team standards, project-specific tasks, shared workflows
- **User prompts** (profile folder) ‚Üí Personal preferences, individual workflows, cross-project utilities

## Mistake 7: No Versioning Strategy for Templates

**Problem**: Teams update templates but don't version them, breaking consistency for files created from older versions.

**‚ùå Bad example:**
````markdown
---
# .github/templates/prompt-template.md
name: [prompt-name]
description: [description]
---
# [Prompt Title]

[Instructions...]
````

No version number, no changelog, no way to know if existing prompts follow current template.

**‚úÖ Solution: Version templates and document changes:**
````markdown
---
# Template Version: 2.1.0
# Last Updated: 2025-12-25
# Changelog:
#   v2.1.0 (2025-12-25): Added 'tools' field for tool restrictions
#   v2.0.0 (2025-11-01): Added 'model' field, removed deprecated 'mode'
#   v1.0.0 (2025-09-15): Initial template
#
# Instructions:
# 1. Copy this file to .github/prompts/
# 2. Rename to [your-prompt-name].prompt.md
# 3. Fill in all [bracketed] fields
# 4. Remove this header comment before committing
---
name: [prompt-name]
description: [Brief description of what this prompt does]
tools: [fetch, search, ...]  # v2.1.0: Optional tool restrictions
model: [Model name]           # v2.0.0: Optional model selection
---

# [Prompt Title]

[Clear description of the task]

## Instructions

1. [Step-by-step instructions]
````

## Key Takeaways

‚úÖ **DO:**
- Separate file types into dedicated folders
- Use consistent naming conventions (kebab-case recommended)
- Follow official file locations and extensions
- Document prompts with README files
- Regularly review and deprecate outdated prompts
- Use workspace prompts for team tasks, user prompts for personal workflows
- Version templates and track changes

‚ùå **DON'T:**
- Mix prompts, snippets, templates, and docs in one folder
- Use inconsistent naming styles across team
- Store files in unrecognized locations or use wrong extensions
- Leave prompts undocumented
- Keep obsolete prompts indefinitely
- Put team-wide prompts in personal user folder
- Update templates without versioning

By avoiding these common mistakes, you'll create a well-organized, discoverable, and maintainable prompt library that serves your entire team effectively.

# üìé Appendix A: Legacy .chatmode.md Migration

**Deprecated as of:** VS Code 1.106  
**Replaced by:** `.agent.md` files in `.github/agents/`

Prior to VS Code 1.106, custom agents were called "chat modes" and used different file conventions:

| Legacy | Current |
|--------|---------|
| `.chatmode.md` extension | `.agent.md` extension |
| `.github/chatmodes/` folder | `.github/agents/` folder |
| Limited frontmatter fields | Extended frontmatter with `handoffs`, `target`, `mcp-servers` |

### Migration Steps

1. VS Code automatically recognizes legacy `.chatmode.md` files as custom agents
2. Open any `.chatmode.md` file and use the **Quick Fix** action (lightbulb) to migrate
3. The Quick Fix renames the file to `.agent.md` and moves it to `.github/agents/`
4. Update any team documentation referencing the old paths

### New Frontmatter Fields

Custom agents now support additional configuration options:

````markdown
---
description: Generate an implementation plan
name: Planner
tools: ['fetch', 'githubRepo', 'search']
model: Claude Sonnet 4
target: vscode  # or 'github-copilot' for remote agents
handoffs:
  - label: Start Implementation
    agent: agent
    prompt: Implement the plan outlined above.
    send: false
mcp-servers:  # For github-copilot target only
  - name: my-server
    url: https://...
---
````
