Agent Skills (<mark>`SKILL.md`</mark>) represent a new paradigm for extending GitHub Copilot with <mark>portable, resource-rich capabilities</mark> that work across multiple platforms.  
Unlike instruction files (which apply silently based on file patterns) or agent files (which define VS Code-specific personas), <mark>skills are on-demand workflows</mark> that include scripts, templates, and examples alongside instructions.

# ğŸ¯ Understanding Skill Files

## What Are Agent Skills?

<mark>**Agent Skills**</mark> are folders containing instructions, scripts, and resources that GitHub Copilot can <mark>**load on-demand to perform specialized tasks**</mark>. They follow an open standard called [agentskills.io](https://agentskills.io/) that enables portability across multiple AI agentsâ€”not just GitHub Copilot.

### Key Characteristics

| Aspect | Description |
|--------|-------------|
| **File** | `SKILL.md` in a dedicated directory |
| **Activation** | <mark>Automatic based on `description` matching your prompt</mark> |
| **Scope** | Task-specific, loaded on-demand |
| **Visibility** | Discoverable - Copilot lists available skills |
| **Portability** | <mark>Cross-platform (VS Code, CLI, GitHub coding agent)</mark> |
| **Resources** | Can include scripts, templates, examples |
| **Standard** | Open standard (agentskills.io) |

## When to Use Skills

**âœ… Use skills for:**

- **<mark>Cross-platform workflows</mark>** - Tasks that should work in VS Code, CLI, and coding agent
- **<mark>Resource-rich tasks</mark>** - Workflows requiring templates, scripts, or examples
- **<mark>Shareable capabilities</mark>** - Functionality you want to contribute to community repositories
- **<mark>Specialized procedures</mark>** - Testing, debugging, deployment workflows with step-by-step guidance
- **<mark>Standardized processes</mark>** - Company-wide procedures that need portable documentation

**âŒ Don't use skills for:**

- **File-specific coding standards** - Use instruction files with `applyTo` patterns
- **Complex tool orchestration** - Use agents with full tool configuration
- **Handoff workflows** - Use agents with handoff capabilities
- **VS Code-specific features** - Use agents or prompts for IDE-specific functionality
- **Quick one-time tasks** - Use chat commands or inline chat

## SKILL.md Format

The `SKILL.md` file is a Markdown document with required YAML frontmatter:

```markdown
---
name: webapp-testing
description: Automated testing workflow for web applications with Jest and React Testing Library. Use when generating test suites, setting up test infrastructure, or debugging test failures.
---

# Web Application Testing Skill

[Skill body with instructions, workflow, and resource references]
```

## YAML Frontmatter Fields

### Required Fields

| Field | Type | Constraints | Example |
|-------|------|-------------|---------|
| `name` | String | <mark>Lowercase, hyphens only, max 64 chars</mark> | `"webapp-testing"` |
| `description` | String | <mark>Max 1024 chars, include use cases</mark> | `"Testing workflow for... Use when..."` |

### Name Field Requirements

```yaml
# âœ… Valid names

# âŒ Invalid names

```

### Description Field Best Practices

The description serves two critical purposes:
1. **<mark>Discovery</mark>** - Copilot shows this when listing available skills
2. **<mark>Matching</mark>** - Copilot uses this to decide when to load the skill

```yaml
# âœ… Good: Specific capabilities AND use cases
description: >
  Automated testing workflow for web applications with Jest and 
  React Testing Library. Use when generating test suites, setting 
  up test infrastructure, or debugging test failures.

# âŒ Bad: Vague, no use case guidance
description: Helps with testing

# âŒ Bad: Too long (exceeds 1024 chars)
description: [500+ word essay about testing philosophy...]
```

**Description Formula:**
```
[What it does] + [Technologies/tools involved] + "Use when" + [specific scenarios]
```

## Core Principles


## Content Length Guidelines

Skills have more flexibility than instruction files because they load on-demand:

| Content Type | Recommended Length | Rationale |
|--------------|-------------------|-----------|
| **Description** | <mark>100-200 chars</mark> | Balance specificity with brevity |
| **SKILL.md body** | <mark>500-1500 words</mark> | Enough detail for complex workflows |
| **Templates** | <mark>20-50 lines each</mark> | Minimal boilerplate, users will expand |
| **Examples** | <mark>30-100 lines each</mark> | Show pattern without full implementation |

### Length Comparison with Other Files

| File Type | Max Recommended | Loading |
|-----------|-----------------|---------|
| **Skill (SKILL.md body)** | <mark>1500 words</mark> | On-demand |
| **Instruction file** | <mark>600 words</mark> | Always (when pattern matches) |
| **Prompt file** | <mark>800 words</mark> | On-demand |
| **Agent file** | <mark>1000 words</mark> | When agent active |

Skills can be longer because they only load when explicitly needed.

# âš ï¸ Critical Limitations and Boundaries

Understanding what skills <mark>**cannot**</mark> do is essential for choosing the right customization type.

## Technical Limitations

### 1. No Tool Control

<mark>Skills **cannot specify which tools are available**</mark> to Copilot. Unlike agents and prompts:

```markdown
<!-- In an agent file, you can specify tools -->
tools:
  - codebase
  - terminal
  - editFiles

<!-- In a skill, you CANNOT do this -->
<!-- Skills rely on whatever tools the current context provides -->
```

**Workaround**: <mark>Suggest tool usage in your instructions</mark>:
```markdown
## Workflow

1. Use the codebase search to find existing test patterns
2. Use the terminal to run the test suite
3. Use file editing to create new test files
```

### 2. No Handoff Capabilities

<mark>Skills cannot orchestrate multi-agent workflows</mark>:

```markdown
<!-- Agent files support handoffs -->
handoffs:
  - agent: security-expert
    when: "security concerns identified"

<!-- Skills do NOT support handoffs -->
<!-- Each skill is a standalone unit -->
```

**When this matters**: If your workflow requires passing control between specialized agents, <mark>use agent files instead</mark>.

### 3. Limited Activation Control

<mark>Activation is based solely on description matching</mark>â€”you cannot:

- Specify file patterns (like `applyTo` in instructions)
- Exclude specific contexts
- Require explicit invocation

```yaml
# âŒ NOT SUPPORTED in skills
applyTo: "**/*.test.js"
excludeAgent: "code-review"
```

### 4. No User Input Variables

<mark>**Skills don't support prompt variables**</mark> like prompt files do:

```markdown
<!-- Prompt files support variables -->
Generate a ${componentType} component named ${name}

<!-- Skills must use generic instructions -->
Generate a component following the patterns below...
```

### 5. Preview Feature Stability

As a preview feature, skills may have:
- Behavior changes between VS Code versions
- Incomplete documentation
- Edge cases not fully handled
- API changes without deprecation warnings

# ğŸš« Common Pitfalls and How to Avoid Them

## Pitfall 1: Vague Descriptions

**Problem**: Generic descriptions cause skills to activate inappropriately or not at all.

**âŒ Bad example:**
```yaml
name: testing
description: Helps with testing
```

This matches too broadly ("test my patience") or not at all (doesn't mention specific technologies).

**âœ… Solution: Be specific with technologies and use cases:**
```yaml
name: react-testing
description: >
  Generate Jest unit tests and React Testing Library integration tests 
  for React components. Use when creating test suites, testing user 
  interactions, or debugging test failures in React applications.
```

## Pitfall 2: Missing Use Cases in Description

**Problem**: Copilot can't determine when to activate the skill.

**âŒ Bad example:**
```yaml
description: Security-focused code review following OWASP guidelines.
```

**âœ… Solution: Include "Use when" trigger phrases:**
```yaml
description: >
  Security-focused code review following OWASP Top 10 guidelines. 
  Use when reviewing pull requests for vulnerabilities, auditing 
  authentication code, or validating input sanitization.
```

## Pitfall 3: External Resource Dependencies

**Problem**: Skill references resources that may not be available or may change.

**âŒ Bad example:**
```markdown
## Templates

Download the test template from https://example.com/templates/test.js
```

**âœ… Solution: Include all resources in the skill directory:**
```markdown
## Templates

Use the [test template](./templates/test.js) included in this skill.
```

## Pitfall 4: Absolute File Paths

**Problem**: Hardcoded paths break when skills are shared or moved.

**âŒ Bad example:**
```markdown
Reference the configuration at /home/user/project/config/jest.config.js
```

**âœ… Solution: Use relative paths from the skill directory:**
```markdown
Reference the [configuration template](./config/jest.config.example.js)
```

## Pitfall 5: Over-Scoped Skills

**Problem**: Skills that try to do everything become too large and unfocused.

**âŒ Bad example:**
```yaml
name: complete-development-workflow
description: >
  Full development workflow including testing, deployment, code review,
  documentation, performance optimization, security auditing, database
  migrations, CI/CD setup, monitoring configuration...
```

**âœ… Solution: Create focused skills with clear boundaries:**
```yaml
# Skill 1: testing-workflow
name: testing-workflow
description: Generate and run test suites...

# Skill 2: deployment-workflow  
name: deployment-workflow
description: Deploy applications to production...

# Skill 3: security-review
name: security-review
description: Review code for security vulnerabilities...
```

## Pitfall 6: Duplicating Built-in Capabilities

**Problem**: Creating skills for things Copilot already does well.

**âŒ Bad example:**
```yaml
name: code-explanation
description: Explain what code does
# Copilot already has /explain command
```

**âœ… Solution: <mark>Focus on specialized, domain-specific workflows</mark>:**
```yaml
name: legacy-code-modernization
description: >
  Analyze legacy code patterns and suggest modern alternatives.
  Use when migrating jQuery to React, upgrading .NET Framework 
  to .NET 8, or converting callback-based code to async/await.
```

## Pitfall 7: No Testing Before Sharing

**Problem**: Skills that work in your environment may fail elsewhere.

**Checklist before sharing:**
- [ ] Test with fresh VS Code profile
- [ ] Verify all relative paths resolve correctly
- [ ] Check resource files are included (not gitignored)
- [ ] Test activation with various prompt phrasings
- [ ] Verify scripts work cross-platform (if applicable)

## How to Test Skills

### 1. Enable Skills in VS Code

First, ensure skills are enabled:

```jsonc
// settings.json
{
  "chat.useAgentSkills": true
}
```

### 2. <mark>Verify Skill Discovery</mark>

Ask Copilot about available skills:

```
User: "What skills are available in this workspace?"
Copilot: [Lists skills with names and descriptions]
```

Verify your skill appears with correct name and description.

### 3. Test Activation with Various Prompts

Test that your skill activates for intended use cases:

```
# Should activate testing skill
"Generate unit tests for this React component"
"Help me debug this failing test"
"Set up Jest for this project"

# Should NOT activate testing skill
"Write documentation for this function"
"Deploy this to production"
```

### 4. Verify Resource Loading

Test that templates and examples load correctly:

```
User: "Use the component test template to test this Button component"
Copilot: [Should reference templates/component.test.js content]
```

### 5. Cross-Platform Testing

If your skill should work across platforms, test in:
- VS Code Copilot Chat
- Copilot CLI (if available)
- GitHub Copilot coding agent (PR workflows)

## Debugging Skills

### Skill Not Activating

1. Check description matches prompt keywords
2. Verify skill is in correct location (`.github/skills/`)
3. Ensure `chat.useAgentSkills` is enabled
4. Check VS Code version (1.107+)

### Resources Not Loading

1. Verify relative paths are correct
2. Check files exist in skill directory
3. Ensure files aren't gitignored
4. Test with explicit resource reference

### Script Execution Issues

1. Check script has execute permissions
2. Verify cross-platform compatibility
3. Review VS Code terminal approval settings

# ğŸ”„ Migration Guidance

## When to Migrate Existing Customizations to Skills

Consider migrating to skills if:

1. **Cross-platform requirement** - Your workflow needs to work in CLI or coding agent
2. **Resource bundling** - You have templates/scripts that should travel with instructions
3. **Community sharing** - You want to contribute to skill repositories
4. **Standardization** - You want to follow the agentskills.io open standard

## Migration Examples

### From Instruction File to Skill

**Before: `.github/instructions/testing.instructions.md`**
```markdown
---
applyTo: "**/*.test.ts"
---
# Testing Standards
Always use Jest with React Testing Library...
```

**After: `.github/skills/testing-workflow/SKILL.md`**
```markdown
---
name: testing-workflow
description: Testing workflow with Jest and RTL. Use when writing tests...
---
# Testing Workflow Skill

## Standards
Always use Jest with React Testing Library...

## Templates
Use [component test template](./templates/component.test.ts)...
```

**Key changes:**
- Add `name` and `description` (remove `applyTo`)
- Create directory structure
- Add templates/examples as files
- Update references to relative paths

### From Prompt File to Skill

**Before: `.github/prompts/generate-tests.prompt.md`**
```markdown
---
description: Generate unit tests for components
---
# Generate Tests
Create comprehensive unit tests for the selected component...
```

**After: `.github/skills/test-generation/SKILL.md`**
```markdown
---
name: test-generation
description: Generate unit tests for React components. Use when...
---
# Test Generation Skill

## Workflow
1. Analyze component structure
2. Use [test template](./templates/component.test.tsx)
3. Cover rendering, interactions, edge cases

## Templates
[Include actual template files]
```

**Key changes:**
- Change YAML fields (`name` required for skills)
- Add templates directory with actual files
- Expand description with "Use when" scenarios

### What NOT to Migrate

**Keep as Instructions:**
- File-specific coding standards (`applyTo` patterns)
- Rules that should always apply silently
- Simple guidelines without resource needs

**Keep as Agents:**
- Complex multi-agent workflows with handoffs
- Tasks requiring specific tool configurations
- VS Code-specific functionality
- Long-running background tasks

**Keep as Prompts:**
- Quick one-time tasks
- Tasks with user input variables
- Simple task definitions without resources

# ğŸ¯ Conclusion

Agent Skills represent <mark>a new paradigm for extending AI coding assistants</mark> with portable, resource-rich capabilities. While currently in preview, they offer unique advantages that complement existing customization options.

**Key takeaways:**

1. **Skills bundle resources with instructions** - Include templates, scripts, and examples alongside guidance
2. **Progressive disclosure optimizes performance** - Only loads what's needed, when needed
3. **Cross-platform portability is the killer feature** - Works across VS Code, CLI, and coding agent
4. **Description quality determines activation success** - Include specific technologies AND "Use when" scenarios
5. **Use relative paths exclusively** - Ensures skills remain portable
6. **Skills complement, don't replace, other customizations** - Use the right tool for each need
7. **Preview status means expect changes** - APIs and behaviors may evolve

### Choosing the Right Customization

| Need | Solution |
|------|----------|
| **Portable workflow with templates** | Skill |
| **File-specific coding standards** | Instruction |
| **One-time task execution** | Prompt |
| **Persistent persona with tools** | Agent |
| **Multi-agent orchestration** | Agent (with handoffs) |

By understanding when to use skills versus other customization types, you can build a powerful, maintainable system that enhances Copilot's effectiveness across all your development workflows.
